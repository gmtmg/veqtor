Claude Code Skills入門ガイド — スラッシュコマンドで作業を自動化

Compass

こんにちは、Compassです。前回のエージェント機能に続いて、今回は「Skills」を紹介します。「エージェントとどう違うの？」という疑問を持っている方も多いと思うので、その違いからていねいに解説しますね。ここから一緒に進みましょう。

Claude Codeを使っていて「この作業、毎回同じ手順を説明しているな」と感じたことはありませんか。Skills（スキル）を使えば、その手順を一度書くだけで、/スキル名 とスラッシュコマンドを打つだけで何度でも再現できます。

この記事では、Skillsをまったく知らない状態から、自分で作って動かすところまでを一歩ずつ解説します。筆者が実際にSkillsを使ってみた所感もお伝えします。

Skillsとは何か

ひとことで言えば、「Claudeに渡すレシピ（手順書）」です。

料理のレシピと同じです。「カレーの作り方」を一度メモしておけば、誰でも何度でも同じ味を再現できますよね。Skillsもまったく同じ考え方で、AIに対して「このタスクはこの手順でやってね」という指示書をファイルで渡します。

Skillsの正体は SKILL.md というテキストファイルです。このファイルを所定のフォルダに置くだけで、Claude Codeが自動的に認識してくれます。

できることの例

  • /fix-issue 123 → GitHub Issue #123を読んで、コードを修正してコミット
  • /deploy staging → ステージング環境へのデプロイ手順を自動実行
  • /review → プロジェクト固有のルールに沿ったコードレビュー
  • /batch "全ファイルのログ形式を統一" → 大規模な一括変更を並列処理

エージェントとSkillsの違い

以前の記事で紹介したエージェント（サブエージェント）と何が違うのか、疑問に思う方も多いでしょう。

ひとことで区別するなら、こうです。

  • Skills = レシピ: メインの会話の中でClaudeが参照する手順書
  • エージェント = 専門家: 独立した作業空間を持つ別のAI担当者

レストランに例えると、Skillsは「レシピブック」で、エージェントは「専門のシェフ」です。シェフは自分のキッチン（独立したコンテキスト）で作業しますが、レシピブックはどのシェフでも参照できます。

使い分けの目安

# Skillsが向いているケース
- プロジェクト固有のコーディング規約を守らせたい
- 毎回同じ手順のタスクをスラッシュコマンドで呼びたい
- チーム全員で同じ作業品質を保ちたい

# エージェントが向いているケース
- 大量のファイルを調査する重い作業を委任したい
- メインの会話を汚さず裏側で作業させたい
- 独立したツール権限で安全に実行したい

迷ったら、まずSkillsを試してみてください。Skillsのほうが軽量で手軽に始められます。

Compass

筆者も最初は「エージェントがあれば十分では？」と思っていました。でも使ってみると、Skillsの手軽さは別格です。この先のハンズオンで、その実感が得られるはず。

Skillsの仕組み — なぜ軽量なのか

Skillsには段階的読み込み（Progressive Disclosure）という賢い仕組みがあります。

  1. 起動時: スキル名と説明文だけを読み込む（1スキルあたり約100トークン）
  2. 呼び出し時: SKILL.md の本文を読み込む
  3. 必要時: 補助ファイル（テンプレートや参考資料）を読み込む

つまり、スキルを100個登録していても、起動時のコンテキスト消費はわずかです。実際に使うスキルだけが詳細に読み込まれます。エージェントのように毎回独立プロセスを起動するわけではないので、応答も速いです。

ハンズオン: 最初のSkillを作ろう

ここからは手を動かしていきます。「コミットメッセージの規約チェッカー」を作ってみましょう。

ステップ1: フォルダを作る

Skillsは .claude/skills/スキル名/ というフォルダ構成で管理します。

# プロジェクトのルートで実行
mkdir -p .claude/skills/commit-check

エージェントの .claude/agents/エージェント名.md とは異なり、Skillsはフォルダの中に SKILL.md を置く形式です。フォルダにすることで、テンプレートや補助ファイルも一緒に管理できます。

ステップ2: SKILL.mdを書く

フォルダの中に SKILL.md ファイルを作ります。

# ファイルを作成して開く（VS Codeの場合）
code .claude/skills/commit-check/SKILL.md

以下の内容を貼り付けて保存してください。

---
name: commit-check
description: コミットメッセージの規約チェック。コミット前や
  コミットメッセージを書くときに使う。
---

コミットメッセージが以下の規約に沿っているかチェックしてください。

## 規約

形式: `type(scope): 説明文`

### typeの種類
- feat: 新機能
- fix: バグ修正
- docs: ドキュメントのみの変更
- style: コードの意味に影響しない変更（空白、フォーマットなど）
- refactor: バグ修正でも新機能でもないコード変更
- test: テストの追加・修正
- chore: ビルドプロセスや補助ツールの変更

### ルール
1. typeは小文字の英語
2. scopeは省略可だがあると望ましい
3. 説明文は日本語OK、50文字以内
4. 本文が必要なら空行を挟んで追記

### 良い例
- feat(auth): ログイン画面にOAuth対応を追加
- fix(api): ユーザー検索のページネーションを修正
- docs: READMEにセットアップ手順を追記

### 悪い例
- Update files（typeなし、説明が曖昧）
- FIX: bug（大文字、説明不足）

ステップ3: 使ってみる

Claude Codeを起動して、スラッシュコマンドで呼び出します。

# Claude Codeを起動
claude

# スラッシュコマンドで呼び出す
/commit-check "Fixed the bug in login"

Claudeがコミットメッセージの規約を読み込み、「Fixed the bug in login は規約に沿っていません。fix(auth): ログインのバグを修正 のように書き直しましょう」と教えてくれます。

Compass

ここまでできましたか？ おめでとうございます！ たった3ステップで最初のSkillが動きました。「思ったより簡単だった」と感じてもらえたら嬉しいです。

また、スラッシュコマンドを使わなくても、「このコミットメッセージをチェックして」と自然に話しかけるだけでClaudeが自動的にこのスキルを見つけて適用してくれます。これが自動発見（Auto-discovery）の仕組みです。

SKILL.mdの構造を理解する

先ほど書いた内容を分解して見てみましょう。

前半: フロントマター（設定）

--- で囲まれた部分がYAML形式の設定です。

---
name: commit-check
description: コミットメッセージの規約チェック。コミット前や
  コミットメッセージを書くときに使う。
---

最低限必要なのはこの2つだけです。

  • name: スキル名。小文字・数字・ハイフンのみ。/名前 でスラッシュコマンドになる
  • description: スキルの説明。「いつ使うか」を含めるのが重要。Claudeはこの説明を見て自動発見する

後半: 本文（指示書）

--- の下に書く部分です。Markdownで自由に書けます。ここにClaudeへの具体的な指示を記述します。

よく使うオプション設定

フロントマターには他にも便利な設定があります。慣れてきたら使ってみてください。

---
name: deploy
description: ステージング環境にデプロイする。
  「デプロイして」と言われたときに使う。
# ↓ ここからオプション
argument-hint: <environment>
disable-model-invocation: true
allowed-tools: Bash(git *) Bash(npm *)
---
  • argument-hint: /deploy と打ったときに表示されるヒント。「何を渡せばいいか」がわかる
  • disable-model-invocation: true: 自動発見を無効化し、/名前 でしか呼べなくする。デプロイのような危険な操作に便利
  • allowed-tools: このスキル実行中に許可するツール。権限を最小限にできる

エージェントとの書き方の違い

すでにエージェント機能を使っている方は、違いが気になるでしょう。並べて比較します。

# エージェント（.claude/agents/code-reviewer.md）
---
name: code-reviewer
description: コードレビューの専門家。
tools: Read, Grep, Glob, Bash     ← エージェント固有
model: sonnet                     ← エージェント固有
---
指示書の本文...


# Skill（.claude/skills/code-review/SKILL.md）
---
name: code-review
description: コードレビューの手順書。
allowed-tools: Read Grep Glob     ← Skill固有（スペース区切り）
---
指示書の本文...

主な違いは3つです。

  1. ファイル配置: エージェントは1ファイル、Skillsはフォルダ+SKILL.md
  2. ツール指定: エージェントは tools（カンマ区切り）、Skillsは allowed-tools（スペース区切り）
  3. 実行方式: エージェントは独立コンテキスト、Skillsはメイン会話内で実行

Skillsの配置場所

Skillsは3つの場所に置けます。優先順位が高い方が上書きします。

# 1. 個人用（すべてのプロジェクトで使える）
~/.claude/skills/スキル名/SKILL.md

# 2. プロジェクト用（チームで共有できる）← おすすめ
プロジェクト/.claude/skills/スキル名/SKILL.md

# 3. 組織用（Enterprise管理）
管理設定から配布

迷ったらプロジェクト用に置きましょう。Gitで共有すれば、チーム全員が同じスキルを使えます。

補助ファイルを追加する

Skillsがフォルダ形式な理由は、補助ファイルを置けるからです。

commit-check/
├── SKILL.md           ← 本体（必須）
├── template.md        ← テンプレート（任意）
├── examples/          ← 具体例（任意）
│   ├── good.md
│   └── bad.md
└── scripts/           ← 補助スクリプト（任意）
    └── validate.sh

SKILL.md の本文から template.md を参照して と書けば、Claudeが必要なタイミングで読み込んでくれます。

すぐに使える組み込みスキル

Claude Codeには最初から使えるスキルが搭載されています。自分で作らなくても、すぐに試せます。

/batch — 大規模な一括変更

/batch src/以下の全コンポーネントにエラーバウンダリを追加

指示を受けると、Claudeが対象ファイルを調査し、5〜30の作業単位に分解します。各単位ごとに並列でエージェントを起動し、それぞれが実装・テスト・コミットまで行います。大量のファイルを変更する作業に威力を発揮します。

/simplify — コード品質レビュー

/simplify src/auth/

最近変更したファイルを対象に、3つの並列エージェントがコードの再利用性・品質・効率性を同時にレビューします。リファクタリングの指針を得たいときに便利です。

/debug — デバッグログの分析

/debug APIレスポンスが空になる

デバッグログを有効にして、セッション中の問題を調査します。エラーの原因を特定するのに役立ちます。

応用テクニック: 動的コンテキスト注入

Skillsの強力な機能の一つが動的コンテキスト注入です。SKILL.mdの中でシェルコマンドを実行し、その結果をClaudeに渡せます。

---
name: pr-review
description: プルリクエストのレビュー。PRレビューを
  依頼されたときに使う。
---

以下のPR情報をもとにレビューしてください。

## PR情報
- 差分: !`gh pr diff`
- コメント: !`gh pr view --comments`
- 変更ファイル: !`gh pr diff --name-only`

## レビュー観点
1. コーディング規約に沿っているか
2. テストが追加されているか
3. セキュリティ上の問題がないか

!`コマンド` と書くと、スキルが呼び出された瞬間にそのコマンドが実行され、結果がClaudeに渡されます。毎回手動でGitHub CLIの出力を貼り付ける手間がなくなります。

引数も使えます。$ARGUMENTS でスラッシュコマンドの引数を受け取れます。

---
name: fix-issue
description: GitHub Issueを修正する。
argument-hint: <issue-number>
disable-model-invocation: true
---

GitHub Issue #$ARGUMENTS を修正してください。

## Issue情報
!`gh issue view $ARGUMENTS`

/fix-issue 42 と打つと、Issue #42の内容が自動で取得されてClaudeに渡されます。

実際に使ってみた所感

ここからは、筆者がSkillsを実際に使ってみて感じたことをお伝えします。

作成は驚くほど簡単だった

最初にSkillsの存在を知ったとき、「エージェントとどう違うのか」「設定が複雑なのでは」と身構えました。しかし、実際に手を動かしてみると、mkdir でフォルダを作り、SKILL.md にMarkdownを書くだけ。最初のスキルは5分もかからず動きました。

特に嬉しかったのは、name と description の2つさえ書けば最低限動くことです。エージェントと同じ感覚で始められます。

自動発見の精度に驚いた

コミットメッセージ規約のスキルを作って、「このメッセージを確認して」と頼んだところ、/commit-check と明示しなくても自動的にスキルが適用されました。description に「コミット前やコミットメッセージを書くときに使う」と書いたおかげです。

逆に、デプロイ用スキルは disable-model-invocation: true にして自動発見を切りました。間違って実行されると困る操作には、このオプションが心強いです。

エージェントとの使い分けが見えてきた

しばらく使ってみて、自分なりの使い分け基準ができました。

  • 「手順を教える」ならSkills: コーディング規約、コミットルール、レビュー観点など、「こうやってね」と伝えるだけのもの
  • 「仕事を任せる」ならエージェント: 50ファイルのリファクタリング、複雑な調査など、独立した作業空間が必要なもの

最初はすべてエージェントで作っていましたが、今は半分以上をSkillsに移行しています。メインの会話内で実行されるため、結果をそのまま会話の流れで確認できるのが便利です。

補助ファイルが地味に便利

フォルダ形式のおかげで、テンプレートファイルや具体例を一緒に管理できます。たとえば、PR作成スキルに template.md を同梱しておくと、毎回同じフォーマットのPR説明文を生成してくれます。

エージェントの場合は1ファイルにすべて詰め込む必要がありましたが、Skillsなら SKILL.md は簡潔に保ちつつ、詳細は別ファイルに分けられます。

注意点: descriptionの書き方で精度が変わる

一つ失敗談もあります。最初に description を「コードを整理するスキル」とだけ書いたところ、意図しないタイミングで発動してしまいました。「いつ使うか」を明確に書くことが本当に大事です。「リファクタリングを依頼されたときに使う」のように具体的なトリガーを含めると、誤発動がなくなりました。

Compass

失敗から学ぶことって多いですよね。descriptionの書き方ひとつで挙動がガラリと変わるのは、最初ちょっと戸惑いました。でも、コツを掴めばすぐ慣れますよ。

うまくいく5つのコツ

コツ1: descriptionに「いつ使うか」を書く

# NG: 曖昧
description: コードに関するスキル

# OK: トリガーが明確
description: コミットメッセージの規約チェック。
  コミット前やコミットメッセージを書くときに使う。

コツ2: 危険な操作は自動発見をオフに

デプロイ、データ削除、本番環境への変更など、間違って実行されたら困るスキルには disable-model-invocation: true を設定しましょう。

コツ3: SKILL.mdは500行以内に

長すぎるとClaudeの処理効率が下がります。詳しい情報は別ファイルに分離して、SKILL.md では概要と手順だけを書きましょう。

コツ4: 既存の知識は書かなくていい

Claudeはすでに多くのプログラミング知識を持っています。「JavaScriptの変数の使い方」のような一般知識は書く必要がありません。プロジェクト固有のルールだけを書くのがポイントです。

コツ5: まず1つ作って試す

完璧なスキルを最初から作ろうとしないでください。最小限の内容で動かしてみて、使いながら改善していくのが一番です。失敗してもファイルを消すだけでやり直せます。

まとめ

この記事のポイントを整理します。

  1. Skills = レシピ。.claude/skills/名前/SKILL.md を置くだけで使える
  2. 必須は2つだけ。name と description さえあれば動く
  3. スラッシュコマンドで呼べる。/スキル名 と打つだけ
  4. 自動発見もできる。descriptionに「いつ使うか」を書けば、自然な会話から自動適用
  5. エージェントとは別物。Skillsは軽量な手順書、エージェントは独立した専門家
  6. 組み込みスキルもある。/batch、/simplify、/debug は今すぐ使える

まずは mkdir -p .claude/skills/my-first-skill でフォルダを作るところから始めてみてください。5分後には最初のスキルが動いているはずです。
